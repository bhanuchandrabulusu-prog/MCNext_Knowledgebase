Starting with the **Summer ’25 release**, Marketing Cloud Next (Marketing Cloud Growth & Advanced Edition) allows you to add an **“Wait Until Event” element** to **form-triggered flows**. This enables the flow to pause until the customer performs a specified action.

For example, consider a flow that sends a **thank-you email** every time a form is submitted. Previously, the flow would immediately proceed to the next action. Now, you can configure it to wait until the customer clicks a link in that email before continuing with the next step.

![]()

> *Although this feature was introduced in the Summer ’25 release, it initially had “*[*known issue*](https://help.salesforce.com/s/issue?id=a028c00000zk0HMAAY)*” and did not function properly. As of September 2025, it is now available for use.*

## **Example Scenario**

One example scenario would be creating a Task for the assigned user — or sending them an email notification — when a customer clicks the “Price List” link included in the thank-you email. This allows marketers to quickly share higher-quality leads with sales representatives from the very beginning of the engagement.

In addition, another **Summer ’25 feature** helps streamline this setup. With the Summer ’25 release, flows can now directly reference the **Lead ID (Salesforce ID)** generated by the **“Create Records”** element.

This means you can directly reference the newly created Lead ID to easily retrieve and leverage fields from that Lead record. As a result, subsequent steps such as creating a Task or sending a notification email can be executed more smoothly.

### **Overall Scenario Flow**

### **Example: Creating a Task for the Assigned User**

In this Task creation example, a **formula resource** named `currectDate_add3Days` has been configured. This formula sets the due date to **3 days after the element is passed**.

- **Assigned To ID**  
   Value: Owner ID (retrieved in the previous Get Records)
- **Name ID**  
   Value: Lead ID (retrieved in the previous Get Records)
- **Status**  
   Value: Not Started (custom value) (retrieved in the previous Get Records)
- **Subject**  
   Value: Check New Lead (fixed/manual input)
- **Due Date Only**  
   Resource Type: Formula  
   Data Type: Date  
   Formula: `{!$Flow.CurrentDate} + 3` (create resource in advance)

### **Example: Configuring a Notification Email for the Assigned User**

If you try to place a **“Send Email” element** directly in a form-triggered flow and activate it, you’ll encounter an “Error” that prevents activation. This is expected behavior under the current specifications, so keep it in mind.

To work around this, place a “**Subflow**” **element** that can launch another flow dedicated to sending the notification email. In this setup, the Subflow calls a separate flow that actually handles the email delivery.

> *Note that Subflow elements can only reference “****Autolaunched Flows”*** *and “****Screen Flows”****.*

Here’s a simple configuration example:

### **Subflow Setup**

1. Click **Autolaunched Flow (No Trigger)**.

2. Once the flow opens, go to the **Toolbox** in the upper left and click **New Resource**.

3. Enter the following details. By checking **Available for input**, the parent flow will be able to pass data into this flow.

- **Resource Type**: Variable
- **API Name**: `getLeadId`
- **Data Type**: Text
- **Availability Outside the Flow**: Check **Available for input**

4. Next, click **Add Element**, select **Get Records**, set the label and API name, and choose **Lead** from Salesforce Objects as the data source.

5. For the Lead record filter, select **Lead ID** on the left and the `getLeadId` variable on the right. Leave other settings as default.

6. Then, click **Add Element** again, search for **Email**, and select **Send Email**.

![]()

7. Set the label and API name. For the **Recipient Address List**, select **Owner ID (User) > Email** from the Lead record retrieved in the previous step.

8. Enter the subject and body of the email. If needed, set **Rich Text Body** to **True**.

9. Once this is configured, name the Autolaunched Flow and activate it. This completes the setup for this flow.

### **Triggering the Subflow**

10. Go back to the original form-triggered flow. In the **Subflow element**, the Autolaunched Flow you just created will appear — select it.

11. Set the label and API name. Change **Set Input Values** from **Not Included** to **Included**.

12. Since the Lead record was retrieved in the prior **Get Records** element, select **Lead ID** from that record.

13. That’s all for the form-triggered flow configuration. With this setup, the notification email will be sent successfully.

![]()

## **Final Note**

Regarding the “**Wait Until Event”** **element** in the **form-triggered flow**: performance may vary depending on your environment. In my case, it sometimes took about **10 minutes** for the flow to trigger after the link was clicked. I recommend being a little patient and allowing some time for it to run.
