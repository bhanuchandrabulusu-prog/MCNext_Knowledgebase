In **Marketing Cloud Next (Marketing Cloud Growth & Advanced Edition)**, I investigated whether there is an equivalent to **Marketing Cloud Engagement’s “Data Views”** (a mechanism that stores historical records such as sends, opens, clicks, bounces, and unsubscribes).

Email engagement data is stored in the following **data streams** and their corresponding **data lake objects**:

- **MessagingEventsEmail-EmailEngagement** (before Summer ’25)
- **MessagingEventsEmailV2-EmailEngagement** (after Summer ’25)

In addition, this data is also stored in the following **data model object**:

- **Email Engagement**

By analyzing these objects, you can perform queries and analysis similar to what is possible with Data Views in Marketing Cloud Engagement.

## About MessagingEventsEmailV2-EmailEngagement

This data lake object records data for the following event types whenever they occur:

- **Sent**
- **Open**
- **Click**
- **Bounce**
- **Unsubscribe**
- **Complaint**

### Key Characteristics

- The data is not tied to a Unified Individual ID but rather linked to the **Individual ID**.
- Data is recorded approximately **2–3 minutes after the event occurs**.
- **Multiple opens** are all retained in the history.
- **Automatic opens** triggered by Apple’s MPP or the Gmail app are also counted as “opens”.
- Information about **Not Sent** events is also recorded.
- Records for **preview sends** and **test sends** are also included.
- When multiple Individual records with the same email address are linked to a Unified Individual record, the Individual Id value used in Email Engagement is taken from one of those linked Individual records. However, as confirmed through an investigation by the Salesforce development team, *“****there is no defined logic that determines which Individual record’s value will be used.****”* My personal understanding is that this behavior varies depending on the identity resolution reconciliation rules.

## Recommended Column Order in Data Explorer

To efficiently search for data, it’s helpful to arrange the following columns in order. **Data Explorer allows a maximum of 10 searchable fields.**

- **IndividualId** — Salesforce ID (18-digit)
- **ContactPointEmailAddress** — Email address
- **EventDateTime** — Event timestamp
- **EventType** — Engagement type
- **SendStatus** — Send status
- **NotSentReason** — Reason for non-delivery
- **BounceType** — Type of bounce
- **BounceReason** — Reason for bounce
- **SendName** — Email content name
- **ResolvedUrl** — Clicked URL

After displaying the data in this order within **Data Explorer**, you can copy the generated **SOQL query** to retrieve results like the following:

```
SELECT  
    IndividualId__c,  
    ContactPointEmailAddress__c,  
    EventDateTime__c,  
    EventType__c,  
    SendStatus__c,  
    NotSentReason__c,  
    BounceType__c,  
    BounceReason__c,  
    SendName__c,  
    ResolvedUrl__c  
FROM  
    MessagingEventsEmailV2_EmailEng_6C0EBACA__dll  
ORDER BY  
    EventDateTime__c DESC  
LIMIT 10
```

## Using Query Editor

By pasting this SOQL query into **Query Editor**, you can perform more detailed searches. While it is possible to write queries manually, **using the query generated by Data Explorer minimizes syntax errors and improves efficiency.**

### Recommended Settings

- **Reduce the displayed records** from the default **100 to around 10** to make the horizontal scroll bar more accessible.
- **Use the wildcard (**`*`**) in the** `SELECT` **clause** to retrieve all available fields in this data lake object.

Example:

```
SELECT *  
FROM   
    MessagingEventsEmailV2_EmailEng_6C0EBACA__dll  
ORDER BY   
    EventDateTime__c DESC  
LIMIT   
    10
```

Additionally, you can **filter data using a** `WHERE` **clause** to refine results. For example, to retrieve **test send** records, use:

```
SELECT   
    IndividualId__c,   
    ContactPointEmailAddress__c,   
    EventDateTime__c,   
    EventType__c,   
    SendStatus__c,   
    NotSentReason__c,   
    BounceType__c,   
    BounceReason__c,   
    ContentName__c,   
    ResolvedUrl__c  
FROM   
    MessagingEventsEmailV2_EmailEng_6C0EBACA__dll  
WHERE   
    IndividualId__c = 'test-send-individual-id'  
ORDER BY   
    EventDateTime__c DESC  
LIMIT   
    10
```

### Important Notes

- **EventDateTime in Query Editor is displayed in UTC.**
- **Unified Individual ID cannot be retrieved** — only the standard Individual ID is available.

## About Email Engagement Data Model Object

You can also retrieve similar information using the **“Email Engagement” data model object (DMO)**, but typically, you cannot obtain content-related information such as the Email Element Name or Email Subject Line. This is because content-related fields are not mapped to **Email Engagement**.

Therefore, it’s necessary to join multiple data model objects to display this information. For reference, try the following query in **Query Editor**:

```
SELECT  
    a.ssot__IndividualId__c AS Id,  
    a.ssot__SendtimeEmailAddress__c AS Email,  
    i.ssot__FirstName__c AS FirstName,  
    i.ssot__LastName__c AS LastName,  
    a.ssot__EngagementDateTm__c + interval '9 hour' AS EngagementDateTime,  
    a.ssot__EngagementChannelActionId__c AS ActionName,  
    a.ssot__EmailRecipientSendStatus__c AS SendStatus,  
    c.ssot__Name__c AS EmailElementAPIName,  
    e.ssot__Name__c AS FlowName,  
    d.ssot__VersionNumber__c AS VersionNumber,  
    h.ssot__Name__c AS SegmentName,  
    a.ssot__EngagementActionReasonText__c AS ActionReason,  
    a.ssot__EmailBounceType__c AS BounceType,  
    a.ssot__BounceReasonText__c AS BounceReason,  
    a.ssot__UnsubscribeSourceText__c AS UnsubscribeSource,  
    a.ssot__ResolvedURL__c AS LinkURL,  
    g.ssot__Subject__c AS EmailSubject,  
    g.ssot__FromAddress__c AS FromAddress,  
    g.ssot__MessagePurpose__c AS MessagePurpose  
FROM ssot__EmailEngagement__dlm a  
JOIN ssot__FlowElementRun__dlm b ON a.ssot__FlowElementRunId__c = b.ssot__Id__c  
JOIN ssot__FlowElement__dlm c ON b.ssot__FlowElementId__c = c.ssot__Id__c  
JOIN ssot__FlowVersion__dlm d ON c.ssot__FlowVersionId__c = d.ssot__Id__c  
JOIN ssot__Flow__dlm e ON d.ssot__FlowId__c = e.ssot__Id__c  
JOIN ssot__BulkEmailMessage__dlm g ON a.ssot__BulkEmailMessageId__c = g.ssot__Id__c  
LEFT OUTER JOIN ssot__MarketSegment__dlm h ON g.ssot__MarketSegmentId__c = h.ssot__Id__c  
LEFT OUTER JOIN ssot__Individual__dlm i ON a.ssot__IndividualId__c = i.ssot__Id__c  
-- WHERE a.ssot__EngagementChannelActionId__c != 'OPEN' /*Action Name Filter*/  
-- WHERE a.ssot__IndividualId__c = '***' /*Individual Id Filter*/  
-- WHERE a.ssot__SendtimeEmailAddress__c = '***' /*Email Filter*/  
-- WHERE i.ssot__FirstName__c = '***' AND i.ssot__LastName__c = '***' /*Name Filter*/  
-- WHERE e.ssot__Name__c = '***' /*Flow Name Filter*/  
-- WHERE h.ssot__Name__c = '***' /*Segment Name Filter*/  
-- WHERE c.ssot__Name__c = '***' /*Email Element API Name Filter*/  
ORDER BY a.ssot__EngagementDateTm__c DESC  
LIMIT 10
```

### Email Engagement Fields

- `ssot__IndividualId__c`: Individual ID
- `ssot__SendtimeEmailAddress__c`: Email address
- `ssot__EngagementDateTm__c`: Engagement date and time (UTC)

\* Added 9 hours to adjust to Japan Standard Time (JST).

- `ssot__EngagementChannelActionId__c`: Type of engagement
- `ssot__EmailRecipientSendStatus__c`: Send status
- `ssot__EngagementActionReasonText__c`: Reason for non-delivery
- `ssot__EmailBounceType__c`: Type of bounce
- `ssot__BounceReasonText__c`: Reason for bounce
- `ssot__UnsubscribeSourceText__c`: Source of unsubscribe
- `ssot__ResolvedURL__c`: Clicked link URL

### Flow Element Run Fields

- None (used for joining)

### Flow Element Fields

- `ssot__Name__c`: API name of the email element

### Flow Version Fields

- `ssot__FlowId__c`: Flow ID
- `ssot__VersionNumber__c`: Flow version number

### Flow Fields

- `ssot__Flow__dlm`: Name of the flow

### Bulk Email Message Fields

- `ssot__Subject__c`: Email subject
- `ssot__FromAddress__c`: Sender email address
- `ssot__MessagePurpose__c`: Commercial or transactional classification

### Market Segment Fields

- `ssot__Name__c`: Name of the segment used

### Individual Fields

- `ssot__FirstName__c`: First name
- `ssot__LastName__c`: Last name
